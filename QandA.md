## Q1. `artifacts/context/` の用途

| 項目 | 内容 |
|---|---|
| 関連仕様 | §6 ディレクトリ構成 |
| 状況 | `artifacts/context/` がディレクトリ構成に記載されているが、仕様書のどこにも用途が定義されていない |
| 質問 | このディレクトリは何を格納するか？（例: Story Agentに渡す文脈ファイル、過去作品の要約キャッシュ、など） |
| **回答** | `artifacts/context/` には当日実行で Story Agent / Review Agent に渡す文脈 bundle を保存する。内容は直近作品要約、`used_themes` スナップショット、文体メモ、レビュー入力キャッシュなどの再生成可能データとする。 |

## Q2. `pending/` にはリンクとコピーのどちらを置くか

| 項目 | 内容 |
|---|---|
| 関連仕様 | §5.2 承認待ち |
| 状況 | 「`stories/` 正本へのリンクまたはコピーを置いてよい」と両案が並記されている |
| 質問 | manual_review 時に `pending/` に置くのはコピーか、シンボリックリンクか、あるいはパスのみ記載したメタJSONか？ |
| **回答** | `pending/` にはコピーやシンボリックリンクを置かず、`stories/` 正本を参照する `*.pending.json` のメタデータだけを置く。JSON には `slug`、正本パス、レビュー要約、作成日時を含める。 |

## Q3. AI生成明記の具体的な表示場所

| 項目 | 内容 |
|---|---|
| 関連仕様 | §21.1 AdSense 必須要件 |
| 状況 | 「AI生成作品である旨を明記する」とあるが表示場所が未定義 |
| 質問 | どこに表示するか？（例: 各記事フッター、サイドバー固定、記事冒頭、フッターグローバル） |
| 補足 | AdSense ポリシー上、全ページで識別可能であることが望ましい |
| **回答** | AI生成明記は全ページ共通フッターに常設する。加えて作品詳細ページではメタ情報直下にも表示し、記事単位でも識別できるようにする。 |

## Q6. Codex CLI 実行失敗時（品質リジェクト以外）の再試行ポリシー

| 項目 | 内容 |
|---|---|
| 関連仕様 | §14.3 再生成 / §19 障害復旧 |
| 状況 | §14.3 の「最大3回」は品質リジェクトの再生成カウント。ネットワーク障害・タイムアウト・CLI クラッシュなどの実行失敗は別カテゴリ |
| 質問 | CLI 実行失敗の場合、再試行回数・待機時間・最終的なフォールバック動作はどうするか？ |
| **回答** | 品質リジェクトとは別カウントで扱う。同一 stage / 同一サブステップ内で 2回まで再試行し、待機時間は `60秒`、`300秒`。それでも失敗した場合は `stage` を維持したまま `result=failed` にしてログ保存と Windows 通知を行い、次回実行で同一 stage から再開する。 |

## Q7. Chirpy テーマへの AdSense 組み込み方法

| 項目 | 内容 |
|---|---|
| 関連仕様 | §3.1 実行環境 / §21 AdSense 要件 |
| 状況 | Chirpy は `_includes/` のオーバーライドで head やレイアウトをカスタマイズできるが、具体的な組み込み箇所が仕様に記載されていない |
| 質問 | AdSense スクリプトをどのファイルに埋め込むか？（例: `_includes/head.html`、`_layouts/post.html`） |
| **回答** | AdSense ローダースクリプトは `site/_includes/head.html` に1回だけ埋め込む。広告枠本体は共通 include 化し、トップ・一覧・詳細を担当する各テンプレートから呼び出す。 |

## Q8. 類似度検査のしきい値

| 項目 | 内容 |
|---|---|
| 関連仕様 | §12.2 類似度検査 |
| 状況 | 「しきい値超過時は差し戻し候補」とあるが、具体的な数値が未定義 |
| 質問 | 3-gram Jaccard のしきい値は何とするか？（一般的な目安: 0.3〜0.5） |
| **回答** | 3-gram Jaccard のしきい値は `0.40` とする。`>= 0.40` は差し戻し候補として Review Agent が詳細確認する。 |

## Q9. review_score の算出方法（各軸の重み付け）

| 項目 | 内容 |
|---|---|
| 関連仕様 | §14.1 評価軸 / §14.2 合格条件 |
| 状況 | 独自性・読みやすさ・一貫性の個別合格ラインは定義されているが、`review_score`（総合）の計算式が未定義 |
| 質問 | 総合スコアは各軸の単純平均か、重み付き平均か？重み配分は？ |
| **回答** | `review_score` は重み付き平均とし、`round(独自性×0.25 + 読みやすさ×0.20 + 一貫性×0.25 + 冒頭の引き×0.15 + 結末の余韻×0.15)` で算出する。禁止事項と AdSense 適合性は加点対象ではなくゲート条件として別判定する。 |

## Q10. Windows 通知の実装方法

| 項目 | 内容 |
|---|---|
| 関連仕様 | §14.3 再生成 |
| 状況 | 「Windows 通知 + ログ保存」とあるが実装方法が未定義 |
| 質問 | Windows 通知は WSL2 から `powershell.exe` を呼び出す方式か、別ツール（例: `notify-send` + WSLg）か？ |
| **回答** | WSL2 から `powershell.exe -NoProfile -File scripts/notify_failure.ps1` を呼び出す方式に統一する。WSLg 依存は持ち込まず、通知失敗時も本処理の失敗ログは必ず残す。 |
